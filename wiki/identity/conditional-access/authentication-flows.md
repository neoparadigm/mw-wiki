# Conditional Access — Authentication Flows

## Conditional Access — Authentication Flows

### Overview
This topic discusses the various authentication flows available in Azure AD Conditional Access (CA), and how they can be used to secure access to resources based on specific conditions. Understanding these authentication flows is crucial for implementing effective CA policies. [New source content: How Multiple Conditional Access Policies Are Applied]

### Key Concepts
- **Multi-Factor Authentication (MFA):** A security mechanism that requires users to provide two or more verification factors to authenticate.
- **Modern Authentication:** A method of authentication that uses modern protocols like OAuth 2.0 and OpenID Connect, as opposed to legacy authentication methods.
- **Device Registration:** The process by which a device is registered with Azure AD, allowing it to be identified and trusted for access to resources.
- **Device Compliance:** A check performed on a device to ensure it meets certain compliance standards before granting access to resources.
- **Conditional Access Policy:** A set of rules that define when and how users can access resources based on specific conditions.
- **Policy Order:** Policies are not applied in any particular order, but all matching policies apply and the resulting access controls required by the policies will be merged [CONFLICT: Daniel Chronlund says policies aren't applied in a specific order, existing entry does not mention this]. However, if both grant and block policies match, block will always win. No exceptions! [New source content]
- **Grant vs Block:** If both grant and block policies match, block will always win. No exceptions! [New source content]
- **Merging Access Controls:** If multiple policies match and they have different Access Controls like Require MFA, Require Compliant Device or Require Azure AD Joined, the requirements will actually be merged and all the access controls from all matching policies have to be fulfilled [New source content].
- **Exporting Policy Assignments:** The Conditional Access Policy Assignment Report is generated by the PowerShell script `Get-ConditionalAccessAssignments.ps1` which exports all CA policy assignments, both group- and user assignments, to an Excel report for easy analysis [New source content: "Export your Conditional Access Policy Assignments to Excel" by Daniel Chronlund].

### Configuration
1. Navigate to the Azure portal and go to Azure Active Directory > Security > Conditional Access.
2. Click on "New policy" to create a new CA policy.
3. Give your policy a name and select the users or groups it should apply to.
4. Under "Cloud apps or actions", choose the apps or actions you want to control with this policy.
5. Under "Conditions", configure the conditions that will trigger this policy, such as device platform, location, client app, etc.
6. Under "Access controls", select the authentication methods and devices that should be required for access [New source content: emphasis added].

Conditional Access is one of Microsoft’s most powerful security features and the central engine for their zero trust architecture. It’s no secret that I love working with Conditional Access and I truly believe that it should be the hearth and soul of every cloud enabled organisations zero trust strategy. But, if you don’t understand how Conditional Access works, it might bring you a false sense of security. In this blog post, I will show you why it is important to understand the Conditional Access policy evaluation process and how to find and exploit flaws in a policy design.

## About Conditional Access

[Conditional Access](https://docs.microsoft.com/en-us/azure/active-directory/conditional-access/overview) is a premium feature of Azure AD and it is disabled by default. There is a feature called [security defaults](https://docs.microsoft.com/en-us/azure/active-directory/fundamentals/concept-fundamentals-security-defaults) that kicks in if you have no policies configured but it is out of scope for this article, and security defaults is always disabled when you have one or more Conditional Access policies in place.

In Conditional Access, everything is allowed by default. And even if you create one or two or three policies, everything you don’t explicitly block is allowed. I’ve put a lot of effort into my [Conditional Access policy design baseline](https://danielchronlund.com/2020/11/26/azure-ad-conditional-access-policy-design-baseline-with-automatic-deployment-support/), so if you are unsure how your policy design holds up, or how to get started, you can use that as a starting point.

In almost every tenant I see out there, there are holes in the Conditional Access design because the designers put too much focus on what use cases they want to allow, not what should be blocked. Conditional Access policies are often designed backwards, and that leaves the tenant vulnerable to attacks.

To educate and raise awareness, I decided to create this guide with examples of how a poorly designed Conditional Access policy design can be exploited to gain access. Use this knowledge for good!

## Passwords and Conditional Access

It’s important to understand that Conditional Access policies in Azure AD are evaluated after the first factor has been approved, namely the user password. I will not go through any password stealing techniques here, but it's crucial to remember that a strong password policy is still essential for securing your tenant.
